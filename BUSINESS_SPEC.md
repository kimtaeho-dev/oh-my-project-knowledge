# 비즈니스 지식 문서 스펙 (Business Spec)

> 이 파일은 .claude/docs/business/ 하위 문서를 생성하거나 갱신할 때 참조하는 **형식 정의서**이다.
> Claude Code는 비즈니스 문서를 생성할 때 반드시 이 스펙의 형식을 따라야 한다.

---

## 공통 규칙

- 모든 문서 상단에 `> 최종 갱신: {YYYY-MM-DD}` 포함
- 기능 ID는 코드 지식 체계의 FEATURES.md와 동일한 `{app}.{기능명_스네이크}` 형식 사용
- 정책 ID 형식: `POL-{NNN}` (도메인 내 순번)
- 에러 ID 형식: `ERR-{NNN}` (도메인 내 순번)
- **최종 문서에 파일 경로, 라인 번호, 변수명, 코드 스니펫을 포함하지 않는다**
- 코드에서 확인되지 않는 규칙은 기재하지 않음. 수동 추가 시 `[수동 추가]` 태그 필수
- 서술 관점: "시스템이 이렇게 동작한다" (사용자/서비스 관점)
- 언어: 비즈니스 용어·규칙 설명은 한글. 기능 ID와 정책/에러 ID만 영문

---

## POLICY_CATALOG.md

### 역할
전체 프로젝트의 비즈니스 규칙을 한눈에 보여주는 최상위 색인. "이 규칙이 어느 도메인에 있는가?"를 즉시 검색하는 용도.

### 형식

```markdown
# 정책 카탈로그

> 최종 갱신: {YYYY-MM-DD}

## 요약

| 도메인 | 정책 수 | 에러 시나리오 수 |
|---|---|---|
| {도메인 한글명} | {N} | {M} |
| **합계** | {총N} | {총M} |

## 도메인별 정책 색인

### {도메인 한글명} ({domain_english_name})

| 정책 ID | 규칙 요약 | 관련 기능 | 분류 |
|---|---|---|---|
| POL-{NNN} | {규칙 한 줄 요약} | {app}.{기능_ID} | {validation / 상태 전이 / 설정값 / 권한 / 데이터 제약} |

(도메인 수만큼 반복)

## 공통 정책 (Cross-Domain)

| 정책 ID | 규칙 요약 | 적용 도메인 | 비고 |
|---|---|---|---|
| COMMON-{NNN} | {규칙 한 줄 요약} | {도메인 목록} | {여러 도메인에 공통 적용되는 이유} |
```

### 작성 규칙
- 분류는 다음 중 선택: validation, 상태 전이, 설정값, 권한, 데이터 제약
- 공통 정책은 2개 이상의 도메인에서 동일한 규칙이 발견된 경우만 등록

---

## domains/{domain-name}/POLICIES.md

### 역할
도메인 내 모든 비즈니스 규칙의 상세 정의. "이 기능에 어떤 규칙이 적용되는가?"를 정확히 파악하는 용도.

### 형식

```markdown
# {도메인 한글명} 도메인 - 서비스 정책서

> 최종 갱신: {YYYY-MM-DD}

---

## {app}.{기능_ID}

### Validation 규칙

#### POL-{NNN}: {규칙 제목}
- **규칙**: {비즈니스 언어로 규칙 설명 — "시스템이 이렇게 동작한다"}
- **위반 시**: {사용자에게 어떤 일이 일어나는가}
- **비즈니스 배경**: {왜 이 규칙이 필요한가 — 확인 가능한 경우만. 불확실하면 생략}

### 상태 정책

#### POL-{NNN}: {상태 전이 규칙 제목}
- **규칙**: {어떤 조건에서 어떤 상태로 전이하는가 — 비즈니스 언어}
- **전이**: {상태 A} → {상태 B}
- **전이 조건**: {사용자/시스템 관점에서 조건 설명}
- **역전이**: {가능 / 불가 — 불가인 경우 이유}
- **영향**: {상태 전이 시 사용자에게 보이는 변화}

### 설정값

#### POL-{NNN}: {설정 제목}
- **규칙**: {무엇이 어떤 값으로 설정되어 있는가}
- **비즈니스 의미**: {이 값이 사용자 경험에 미치는 영향}
- **변경 시 영향**: {이 값을 변경하면 사용자에게 어떤 변화가 생기는가}

### 에러 시나리오

#### ERR-{NNN}: {에러 제목}
- **메시지**: "{실제 에러 메시지 텍스트}"
- **발생 상황**: {비즈니스 언어로 상황 설명}
- **사용자 영향**: {사용자에게 어떻게 보이는가}
- **대응 방법**: {사용자가 이 에러를 만났을 때 어떻게 해야 하는가}

---

(기능 수만큼 반복)
```

### 작성 규칙
- 섹션 구조는 FEATURES.md의 기능 ID를 그대로 사용 (두 체계의 연결 고리)
- 모든 서술은 "시스템이 ~한다", "사용자가 ~하면" 형태의 비즈니스 언어로 작성
- 비즈니스 배경은 코드/주석에서 추론 가능한 경우만 기술. 불확실하면 생략하거나 `[수동 추가]`로 나중에 보강
- 에러 시나리오의 "대응 방법"은 필수 — 이 문서를 CS 담당자가 참조할 수 있다
- `[수동 추가]` 태그가 붙은 내용은 자동 갱신 시 보존된다

## domains/{domain-name}/SPEC.md

### 역할
도메인의 각 기능을 비개발자가 이해할 수 있는 기획서 형태로 기술. "이 기능은 왜 이렇게 동작하는가?"에 답하는 문서.

### 형식

```markdown
# {도메인 한글명} 도메인 - 기획서

> 최종 갱신: {YYYY-MM-DD}

## 도메인 개요

{이 도메인이 서비스에서 담당하는 역할을 2~3문장 설명}

---

## {기능 한글명}

### 이 기능은 무엇인가
{기능의 목적과 동작을 사용자 관점에서 설명}

### 사용자 시나리오
1. {사용자가 ~을 한다}
2. {시스템이 ~을 수행한다}
3. {결과로 ~이 표시/저장된다}

### 왜 이렇게 동작하는가
- **{설계 결정 1}**: {이유}
- **{설계 결정 2}**: {이유}

### 주요 정책
- POL-{NNN}: {정책 한 줄 요약} (상세: POLICIES.md)

### 에러 상황
- ERR-{NNN}: {에러 한 줄 요약} — {사용자 대응 방법}

### 관련 용어
- {용어}: {간략 정의} (상세: GLOSSARY.md)

---

(기능 수만큼 반복)
```

### 작성 규칙
- **코드 경로, 변수명, 기술 용어를 사용하지 않는다**
- "왜 이렇게 동작하는가" 섹션은 주석/설정값/코드 구조에서 추론 가능한 것만 기술. 불확실하면 `[확인 필요]` 명시
- 사용자 시나리오는 SERVICE_FLOW.md의 흐름을 사용자 관점으로 재서술
- 정책/에러는 ID만 참조하고 상세는 POLICIES.md로 위임

---

## domains/{domain-name}/GLOSSARY.md

### 역할
도메인에서 사용되는 용어의 정의. 같은 개념을 다른 이름으로 부르는 혼란을 방지.

### 형식

```markdown
# {도메인 한글명} 도메인 - 용어집

> 최종 갱신: {YYYY-MM-DD}

## {가나다 순}

### {한글 용어}
- **정의**: {이 프로젝트에서의 의미}
- **혼동 주의**: {다른 용어와의 차이, 또는 일반적 의미와의 차이}
- **예시**: {구체적 값이나 상황 예시}

(용어 수만큼 반복)
```

### 작성 규칙
- 용어는 가나다 순 정렬
- "혼동 주의"는 실제로 혼동이 발생할 수 있는 경우만 기술
- 코드 변수명/필드명을 용어 정의에 포함하지 않는다
- 너무 일반적인 용어는 프로젝트 특화 의미가 있는 경우만 등록

---

## GLOSSARY.md (전체)

### 역할
모든 도메인의 용어를 통합한 프로젝트 전체 용어집.

### 형식

```markdown
# 전체 용어집

> 최종 갱신: {YYYY-MM-DD}

## {가나다 순}

### {한글 용어}
- **정의**: {이 프로젝트에서의 의미}
- **소속 도메인**: {도메인 한글명}
- **혼동 주의**: {다른 용어와의 차이}

(용어 수만큼 반복)

## 정의 불일치 목록

| 용어 | 도메인 A 정의 | 도메인 B 정의 | 상태 |
|---|---|---|---|
| {용어} | {정의 A} | {정의 B} | {미해소 / 해소 — 통일 정의} |
```

### 작성 규칙
- 도메인별 GLOSSARY.md에서 자동 취합
- 동일 용어가 여러 도메인에서 다르게 정의된 경우 "정의 불일치 목록"에 기록
